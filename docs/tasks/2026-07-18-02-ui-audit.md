# TASK: UI Audit — Compare FrameSheet UI with MoviePrint

## Goal

Produce a gap analysis between FrameSheet's current UI and MoviePrint
(fakob/MoviePrint_v004), which is the visual reference this app should
converge toward. **This task is audit-only. Do not implement any UI
changes.**

## Output

Write the audit to `docs/UI_AUDIT.md`.

## Reference material

- Screenshots of MoviePrint are in `docs/reference/` (added by Master).
  Treat these as the visual ground truth.
- If helpful, consult the MoviePrint source on GitHub
  (fakob/MoviePrint_v004, Electron/React). Do not port code; use it only
  to understand layout structure, settings taxonomy, and parameter
  ranges/defaults.

## Steps

### 1. Inventory current FrameSheet UI

- Map the SwiftUI view hierarchy: every view/file, its responsibility,
  and parent-child relationships.
- List all user-facing settings/controls (name, type, range, default,
  where it lives in the UI).
- Note how state flows (e.g. @State/@Observable/bindings) and where
  ffmpeg invocation is triggered relative to the UI layer.

### 2. Inventory MoviePrint UI (from screenshots + source)

- Overall layout: panels, header, preview grid, footer.
- Settings taxonomy: grid (rows × columns), margins, corner radius,
  frame/header metadata display, output options, appearance (dark UI),
  and anything else visible.
- Interaction model: live preview updates, drag/drop, in/out points,
  per-thumbnail actions — note which exist even if out of scope for now.

### 3. Gap analysis

For each area, produce a table with:

| Area | MoviePrint | FrameSheet current | Gap | Effort (S/M/L) | Notes |

Areas to cover at minimum: layout structure, settings panel, preview
grid, header/metadata rendering, appearance/theming, live preview
behavior.

### 4. Refactoring readiness assessment

The project originated as a single `main.swift` build. Assess whether
the current view decomposition can absorb a MoviePrint-style layout
(sidebar settings panel + live preview grid) without major rework:

- Which views need splitting or extraction?
- Is rendering logic (contact sheet composition) separable from view
  code?
- Any state management changes required for live preview?

List concrete refactoring prerequisites, ordered.

### 5. Recommendation

End `docs/UI_AUDIT.md` with a proposed phased plan (Phase 1/2/3) for
converging on MoviePrint's UI, so the follow-up implementation TASK.md
can be derived from it directly.

## Constraints

- No code changes in this task (docs only).
- Follow repo conventions in CLAUDE.md / AGENTS.md.
- Build nothing; static analysis of the codebase is sufficient.
